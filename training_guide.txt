================================================================================
 LABEL ENCODING
================================================================================

CSV value → tensor encoding:

  CSV value  Meaning                          Tensor value
  ---------  -------------------------------- ------------
      1      Pathology positively mentioned   1.0  (positive)
      0      Pathology explicitly ruled out   -1.0 (negative)
     NaN     Label not mentioned in report    -1.0 (negative — same as 0)
     -1      Ambiguous / uncertain mention    0.0  (ignored)

NaN and 0 are both treated as -1.0 because a finding absent from the report
is functionally the same as one explicitly ruled out: we have no evidence it is
present. Only -1 (uncertain language such as "possible" or "cannot exclude") is
treated as ambiguous and ignored — it provides no clean learning signal either way.

Consequence for conflict detection: if sample A has Edema=1 and sample B has
Edema=NaN, they will be flagged as a conflict in negative_aware mode (one
confirms, the other provides no-evidence-of-presence, treated as denial).


================================================================================
 KNOB 1 — CAPTION MODE  (--caption-mode)
================================================================================

Controls the text caption paired with each image during training.
Captions are built from the image's POSITIVE (value=1) labels only.

  single   One random positive label         "atelectasis"
  pair     Two random positive labels        "fracture and edema"
  both     Randomly single or pair           (DEFAULT)
           (75 % single, 25 % pair — weighted toward single to match eval distribution)

Recommendation: keep --caption-mode both (default). Eval uses both single and
pair query types, so training on a mix generalises better.


================================================================================
 KNOB 2 — MATCH MODE  (--match-mode)
================================================================================

Controls which (image, text) pairs in the batch are treated as positives.

WHY THIS MATTERS
----------------
Standard CLIP treats each (image_i, text_i) as the one and only positive for
sample i, pushing every other sample away as a negative. In a multi-label
medical dataset this is wrong: if image A and image B both show "Edema and
Pleural Effusion", pulling image A away from text B is a harmful training signal.
It gets more subtle with partial overlap: two images may share "Cardiomegaly"
but not "Atelectasis", so they are not perfect matches — they should be
attracted along the shared label but not unconditionally. That is exactly what
the soft-target label-aware modes handle (threshold controls strictness).
The label-aware modes fix this by identifying which pairs share pathology labels.

MODES
-----
  standard
    Diagonal-only pairs (vanilla CLIP). Each sample is its own positive.
    Uses open_clip's ClipLoss with full distributed all-gather. Good baseline.

  single_label
    Two samples are a positive pair if they share ≥1 label with value 1.
    The loss target is a soft matrix: probability mass is split equally across
    all in-batch positives so each row still sums to 1.
    Example: image with {Edema} matches text built from {Edema, Atelectasis}.

  two_label
    Same as single_label but threshold is ≥2 shared positive labels.
    More selective — reduces noise from coincidentally shared single labels.
    Best paired with --caption-mode pair (captions name two labels).

  negative_aware
    Builds on single_label (threshold ≥1) and adds explicit REPULSION for
    conflict pairs. A conflict = one sample has a label encoded as 1.0 and
    the other has the same label encoded as -1.0 (i.e., CSV 0 or NaN).
    Conflicting pairs are pushed apart with a hinge loss (see Knob 3).
    Extra flags:
      --negative-weight λ  weight for repulsion term  (default 0.5)
      --negative-margin τ  cosine-sim threshold; pairs below τ are not penalised
                           (default 0.0 — penalise any positive similarity)

Diagnostics logged at step 0 of each epoch (label-aware modes only):
  avg_positives_per_sample  — should be >1.0; if ≈1.0 the batch is too small
                              or labels are too sparse (increase --batch-size)
  diagonal_only (%)         — samples with no in-batch positive partner; keep <20 %
  conflict_pairs (%)        — pairs being repelled (negative_aware only)


================================================================================
 KNOB 3 — LOSS VARIANT  (--loss)
================================================================================

Controls the mathematical form of the contrastive loss. Orthogonal to --match-mode:
any combination of the two is valid.

  clip    (DEFAULT) Softmax multi-positive cross-entropy.
          Each row of the B×B similarity matrix is normalised via softmax.
          For matching pairs the loss distributes probability mass across all
          in-batch positives. Non-matching pairs are implicit negatives via
          the softmax denominator.
          Formula (label-aware):
            target[i,j] = (shared ≥k labels AND no conflict) / row_sum
            L_i2t = -Σ_j target[i,j] · log_softmax(logits)[i,j]
            L_t2i = same with logits transposed
            L_clip = (L_i2t + L_t2i) / 2
            L_total = L_clip + λ · relu(sim[conflict] - τ).mean()  [negative_aware]

  siglip  Sigmoid binary cross-entropy, one independent term per pair.
          No softmax denominator — each (image_i, text_j) is a binary decision:
            target[i,j] = +1 if positive pair, −1 if negative/conflicting
            loss[i,j]   = −logsigmoid(target[i,j] · logits[i,j])
            L_siglip    = sum over all B² pairs / B
          Key advantage: loss per pair is decoupled from batch size.
          Conflicting pairs already receive target −1 so the SigLIP loss
          provides per-pair repulsion without a separate hinge. The optional
          hinge is still added in negative_aware mode for extra emphasis.
          Note: SigLIP model checkpoints (e.g. ViT-B-16-SigLIP) carry a
          learned logit_bias; vanilla CLIP models don't — both work fine.

Tuning λ (--negative-weight) and τ (--negative-margin):
  λ = 0.5   balanced start
  λ = 0.25  if training loss is unstable
  λ = 1.0   aggressive repulsion
  τ = 0.0   penalise any positive cosine similarity between conflict pairs
  τ = -0.2  more lenient (only penalise if already fairly similar)
  τ = 0.2   more aggressive (push apart even if similarity is low)

 RECIPES
================================================================================

  Recipe A — Baseline (vanilla CLIP loss, diagonal pairs)
    python train_lora.py ... --match-mode standard --loss clip

  Recipe B — Soft positives (single label match, CLIP loss)
    python train_lora.py ... --match-mode single_label

  Recipe C — Strict positives (two shared labels)
    python train_lora.py ... --match-mode two_label --caption-mode pair

  Recipe D — Full negative-aware CLIP (attract similar, repel conflicting)
    python train_lora.py ... --match-mode negative_aware \
        --negative-weight 0.5 --negative-margin 0.0

  Recipe E — SigLIP loss, label-aware (per-pair binary signal)
    python train_lora.py ... --loss siglip --match-mode single_label

  Recipe F — SigLIP + negative-aware (strongest repulsion signal)
    python train_lora.py ... --loss siglip --match-mode negative_aware \
        --negative-weight 0.5 --negative-margin 0.0


================================================================================
 DESIGN NOTES
================================================================================

Label-aware modes operate on the LOCAL batch only (no cross-GPU label
gathering). For distributed runs, positive mining is per-GPU. Standard mode
retains full distributed all-gather via open_clip's ClipLoss / SigLipLoss.

Conflict detection covers both explicit negatives (CSV 0) and missing labels
(NaN), since both are encoded as -1.0. This means a pair where one report
confirms a finding and the other simply does not mention it will be flagged
as conflicting in negative_aware mode. Whether this is desirable depends on
the assumption about absence of evidence — currently we assume it equals
evidence of absence.

Uncertain labels (CSV -1, encoded as 0.0) are fully ignored: they contribute
neither to positive overlap counting nor to conflict detection.

Possible future extension: "shared-negative" positives — pairs where both
reports explicitly deny the same label (both have CSV 0 for Edema) could be
treated as an additional positive-overlap channel, since both images likely
show the same finding-free appearance for that pathology.

    ≈ 1.0  → batch too small or mode too strict; increase --batch-size or switch
             from two_label to single_label
    > 2.0  → healthy; label-aware modes have real signal to exploit

  diagonal_only
    > 20%  → too many samples have no in-batch partner; same remedies as above

  conflict_pairs
    < 1%   → very few conflicting pairs; --negative-weight has little effect
    5-15%  → healthy signal for repulsion term in negative_aware mode

Training/val loss:
  val_loss rises while train_loss falls → overfitting; reduce --lr or --lora-r,
  or add more dropout (--lora-dropout 0.1)
  val_loss plateaus quickly → learning rate too high or LoRA rank too low
